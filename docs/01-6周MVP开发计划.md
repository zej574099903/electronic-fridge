# 电子冰箱 App：6 周 MVP 开发计划

## 1. 项目目标

在 6 周内完成一个可上线测试的 `MVP`，用于验证以下核心假设：

- 用户愿意记录冰箱中的物品
- 用户对“临期提醒”有真实需求
- 用户对“今天优先吃什么”的任务视图有感知价值
- “剩菜模式”是否能成为高频入口

本阶段不追求做成完整平台，而是验证最小闭环：

- 录入
- 存储
- 提醒
- 消耗建议

## 2. MVP 范围

### 2.1 本期必须完成

- 用户登录
- 冰箱物品管理
- 分类管理
- 存放分区管理
- 剩菜快捷录入
- 临期提醒
- 首页任务看板
- 简单家庭共享（可选加入同一家庭）
- 基础设置页
- 管理后台最小版本

### 2.2 本期不做

- 复杂 AI 拍照识别
- 小票识别
- 电商订单导入
- 支付与会员系统
- 智能菜谱推荐引擎
- 复杂埋点平台
- 多端复杂同步策略
- 精细化权限系统

## 3. 建议技术方案

### 3.1 客户端

- `React Native + Expo`
- 状态管理：`Zustand`
- 请求库：`Axios`
- 表单：`React Hook Form`
- 页面导航：`Expo Router` 或 `React Navigation`

### 3.2 服务端

- `Next.js`
- API：`Route Handlers` 或 `API Routes`
- 鉴权：`NextAuth` 或自定义 JWT

### 3.3 数据层

- `MongoDB Atlas`
- `Mongoose`
- 缓存/任务：前期可先不引入 `Redis`，提醒先走简化方案

### 3.4 后台

- `React + Ant Design`

### 3.5 推送与消息

- 先做 App 内提醒列表
- 第二阶段接 `Expo Push Notifications`

## 4. 6 周里程碑

## Week 1：产品冻结 + 项目脚手架

### 目标

完成需求冻结和基础工程初始化。

### 本周交付

- 明确 MVP 边界
- 输出页面清单
- 输出数据模型初稿
- 创建移动端项目
- 创建后端项目
- 创建后台项目
- 建立开发分支规范
- 建立 UI 基础组件规范

### 任务拆分

- 产出 PRD Lite
- 明确物品分类体系
- 明确提醒规则
- 定义接口返回格式
- 创建项目目录结构
- 接入 MongoDB Atlas
- 跑通登录鉴权 Demo

### 验收标准

- App 能启动
- 后端能连库
- 能完成注册/登录的最小流程
- 文档齐全，可进入功能开发

## Week 2：核心数据闭环

### 目标

完成“用户-家庭-冰箱空间-物品”的核心数据结构与 CRUD。

### 本周交付

- 用户资料接口
- 家庭空间模型
- 冰箱分区模型
- 物品增删改查
- 分类与默认规则

### 任务拆分

- 完成 `users`、`households`、`storageSpaces`、`items` 集合设计
- 实现物品列表页
- 实现新增物品页
- 实现编辑物品页
- 实现删除与状态变更
- 实现分类筛选与位置筛选

### 验收标准

- 用户可新增、查看、编辑、删除物品
- 物品有分类、到期时间、位置等字段
- 数据结构可支撑后续提醒与首页看板

## Week 3：剩菜模式 + 首页任务看板

### 目标

做出产品差异化能力，让首页不是“数据库列表”，而是“行动页面”。

### 本周交付

- 剩菜快捷录入
- 首页任务看板
- 今日优先吃模块
- 48 小时内到期模块
- 最近新增模块

### 任务拆分

- 设计剩菜录入专属表单
- 设计默认提醒时间策略
- 设计首页聚合接口
- 完成首页卡片 UI
- 完成任务状态交互：已吃掉 / 已丢弃 / 稍后处理

### 验收标准

- 用户打开首页能看到明确行动建议
- 剩菜录入路径比普通录入更快
- 首页聚合逻辑正确

## Week 4：提醒系统 + 设置页

### 目标

让核心价值真正可用。

### 本周交付

- 到期提醒任务
- App 内提醒中心
- 用户提醒偏好设置
- 剩菜提醒规则
- 过期状态自动更新

### 任务拆分

- 编写每日检查临期物品任务
- 建立 `notifications` 集合
- 首页与提醒页联动
- 设置页面支持提醒开关
- 后端支持“明天提醒我”“提前 1 天提醒”之类规则

### 验收标准

- 系统可生成临期提醒
- 用户能查看提醒记录
- 提醒不会重复失控生成

## Week 5：家庭共享 + 管理后台最小版

### 目标

完成最小协作能力和后台支撑。

### 本周交付

- 家庭加入/邀请
- 家庭共享物品列表
- 操作记录
- 后台登录
- 分类规则后台
- 用户反馈后台

### 任务拆分

- 支持创建家庭
- 支持加入家庭
- 家庭成员共享查看物品
- 记录谁新增/删除/吃掉物品
- 完成管理后台基础表格页面
- 完成默认保鲜规则配置

### 验收标准

- 两个账号能加入同一家庭并共享数据
- 后台能维护默认分类和规则

## Week 6：测试、优化、灰度发布

### 目标

完成一轮真实用户可测试版本。

### 本周交付

- Bug 修复
- UI 收口
- 基础埋点
- 测试账号
- 种子用户测试版本
- 上线准备文档

### 任务拆分

- 核心路径走查：登录、录入、提醒、已吃掉
- 性能优化
- 空状态与异常态补齐
- 准备 TestFlight / Android 内测包
- 邀请 10-20 位种子用户试用
- 回收反馈表

### 验收标准

- 用户能独立完成主要路径
- 无阻断级 Bug
- 可进行真实用户测试

- 拆任务

### 周二到周四

- 功能开发
- 接口联调

### 周五

- 测试与修复
- 补文档

### 周六

- 复盘 + 准备下周

## 7. 风险控制

### 最大风险

- 范围膨胀
- 提醒逻辑复杂化
- 页面过多导致延期
- 过早引入 AI 识别

### 控制原则

- 每周只围绕一个里程碑推进
- 所有“想加的功能”先放入 Backlog
- 本期不因识别能力拖延主流程

## 8. 6 周后成功标准

如果 6 周后满足以下条件，就说明 MVP 有意义：

- 有 10-20 个种子用户开始使用
- 至少一半用户完成过 3 次以上录入
- 有用户明确表达“提醒很有用”或“首页先吃什么有帮助”
- 剩菜模式有明显使用率

## 9. 6 周后下一步

如果 MVP 验证通过，再进入第二阶段：

- 拍照辅助录入
- 简单 AI 识别
- 菜谱推荐增强
- 推送通知增强
- 数据分析看板
- 会员能力探索
